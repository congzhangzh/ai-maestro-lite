# 设计方案

## 总体架构

系统分成三层：

- 桌面客户端：Electron + React + Ant Design
- 后端服务：Node.js + Fastify + SQLite
- 共享类型层：统一接口、场景 ID、探针和审计结构

## 桌面端设计

桌面端首页固定展示 5 个场景，不做动态场景市场。

每个场景只暴露 4 个动作：

- 安装
- 更新
- 启动
- 自检

补充设置按场景内聚：

- VS Code：内部加速代理模板
- Claude Code：模型切换、手动模型输入、跳过可选配置步骤
- GitHub 浏览器：受控访问和 URL 审计查看

## 后端设计

后端服务固定提供：

- 设备注册
- 场景信息下发
- 更新检查
- 浏览器策略下发
- Claude 网关配置下发
- VS Code 代理模板下发
- 动作探针上报
- URL 审计上报

后端数据存储统一放在 SQLite 中，默认路径为：

- `data/app-store.sqlite`

当前表职责：

- `devices`：设备注册
- `probes`：动作探针
- `audits`：URL 审计
- `bitable_rows`：飞书多维表格同步结果，包含来源记录 ID、表 ID 和外部 record id
- `notifications`：告警与通知

## 更新设计

更新机制采用“默认值 + 配置覆盖”方式：

- 代码中内置默认版本、默认说明、默认下载地址
- 若存在 `data/runtime-config.json`，则使用其中的配置覆盖默认值
- 也支持通过 `AI_MAESTRO_RUNTIME_CONFIG` 指定自定义配置路径

这样可以同时满足：

- 开发环境不配配置也能直接运行
- 企业部署时可按环境、网络和发布策略覆盖

## 审计设计

系统有两类审计：

- 动作探针：进入场景、安装、更新、启动、自检
- URL 审计：受控 GitHub 浏览器中的网络请求

GitHub 浏览器的原则是：

- 只允许 GitHub 白名单域名
- 强制走公司代理
- 记录放行和拦截的请求
- 审计数据回传后端，再由后端同步到中台/飞书

## 飞书同步设计

飞书同步的执行顺序固定为：

1. 客户端把探针和 URL 审计写入后端数据库
2. 后端 worker 从 SQLite 中取出未同步数据
3. worker 调用飞书开放 API，把数据写入飞书多维表格
4. 写入成功后，后端把同步结果再次记入 SQLite 的 `bitable_rows`
5. 原始 `probes` / `audits` 记录被标记为已同步

飞书相关信息通过环境变量控制，不硬编码在仓库中。
